## 开发反思

| 日期 | 反思 | 来源 |
|------|------|------|
| 2026-07-14 | **发布工作区残留**：v1.0.0 发布后，根目录 `aikun-releases/`（含发布包 + README）未及时清理，导致自动归类时反复识别为「越界重复目录」，需要额外的手动确认和清理。教训：发布后立即删除临时发布工作区，或者直接在项目 `release/` 目录内完成发布，不创建根级散落目录 | 每日工作区自动归类 |
| 2026-07-14 | **项目外发布包冗余**：发布包部分存在于 `projects/aikun-gui/release/` 中，部分散落在 `_temp/工作区残留/` 和原 `aikun-releases/`，管理混乱。教训：发布流程应统一为「构建 → 验证 → 复制到 `release/` → 上传 GitHub → 清理构建中间件」，不跨目录多重备份 | 每日工作区自动归类 |

---

### 归类经验

- `release/` 目录是发布包的单一制高点，所有 `.exe`/`.zip` 只进不出
- 发布完成后，构建中间件（`build/`、`dist/`）可删除以节省空间
- 项目根级 README 与发布版 README 应保持同步
